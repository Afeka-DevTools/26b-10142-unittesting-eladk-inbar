# Java Unit Testing Guide: JUnit 5

---

## 1. The Core Philosophy: Arrange, Act, Assert (AAA)

Every well-written unit test in Java follows a strict three-step structure. Keeping these steps visually separated in your code makes your tests easy to read and debug.

* **Arrange:** Set up the exact conditions required for the test. Initialize objects, define variables, and prepare the environment.
* **Act:** Call the specific Java method you are trying to test.
* **Assert:** Verify that the method's output (or the resulting state of the object) matches your expectations using JUnit assertion methods.

---

## 2. The Target Code (What We Are Testing)

To demonstrate, let’s imagine you are writing software for a logistics center. You have a simple Java class responsible for tracking the number of pallets in a specific warehouse zone. 

Here is the source code we need to test:

```java
public class InventoryTracker {
    private int stockLevel;

    public InventoryTracker(int initialStock) {
        this.stockLevel = initialStock;
    }

    // Method to test: Adding new pallets to the zone
    public void addPallets(int count) {
        if (count < 0) {
            throw new IllegalArgumentException("Cannot add a negative number of pallets.");
        }
        this.stockLevel += count;
    }

    public int getStockLevel() {
        return this.stockLevel;
    }
}
```

---

## 3. Writing the Unit Tests (JUnit 5)

Test classes are typically created in a dedicated `src/test/java` directory. We will use **JUnit 5**, which relies heavily on annotations like `@Test` and static methods from the `Assertions` class.

### Example A: Testing the "Happy Path"
The "happy path" is when the code is used exactly as intended with valid inputs.

```java
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

public class InventoryTrackerTest {

    @Test
    public void testAddPalletsIncreasesStockCorrectly() {
        // 1. ARRANGE
        // Start with 10 pallets in the zone
        InventoryTracker tracker = new InventoryTracker(10); 
        int newPalletsArrived = 5;

        // 2. ACT
        // Process the new arrival
        tracker.addPallets(newPalletsArrived); 

        // 3. ASSERT
        // We expect the new stock level to be 15
        assertEquals(15, tracker.getStockLevel(), "Stock level should increase by 5.");
    }
}
```

### Example B: Testing the "Unhappy Path" (Edge Cases)
You must also prove that your code handles bad data correctly. In our `InventoryTracker`, trying to add a negative number of pallets should trigger an `IllegalArgumentException`. 

In JUnit 5, you test for exceptions using `assertThrows()`.

```java
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

public class InventoryTrackerTest {

    @Test
    public void testAddNegativePalletsThrowsException() {
        // 1. ARRANGE
        InventoryTracker tracker = new InventoryTracker(10);
        int invalidPalletCount = -3;

        // 2. ACT & ASSERT
        // For exceptions, the Act and Assert steps are combined inside a lambda expression.
        Exception exception = assertThrows(IllegalArgumentException.class, () -> {
            tracker.addPallets(invalidPalletCount);
        });

        // Optional: Verify the exception contains the correct error message
        assertEquals("Cannot add a negative number of pallets.", exception.getMessage());
    }
}
```

---

## 4. Essential JUnit Assertions Glossary

Here are the most common assertion methods you will use when writing Java tests:

* **`assertEquals(expected, actual)`**: Checks if two values are equal. Always put the expected value first.
* **`assertTrue(condition)`**: Checks if a boolean condition evaluates to true.
* **`assertFalse(condition)`**: Checks if a boolean condition evaluates to false.
* **`assertNull(object)`**: Verifies that an object reference is null.
* **`assertNotNull(object)`**: Verifies that an object reference is not null.
* **`assertThrows(Exception.class, () -> { ... })`**: Verifies that the code block inside the lambda throws the specified exception.

---

## 5. Golden Rules for Writing Good Tests

1. **Test One Thing per Method:** A single `@Test` method should generally contain only one logical assertion (or a few closely related ones). If a test fails, you should know exactly what broke without having to read through lines of unrelated assertions.
2. **Keep Tests Independent:** Tests should never rely on each other. If `testA` must run before `testB` for `testB` to pass, your tests are flawed. JUnit does not guarantee the execution order of tests.
3. **Name Tests Descriptively:** A test name should explain exactly what is being tested and what the expected outcome is (e.g., `testAddNegativePalletsThrowsException` is much better than `testAddPallets2`).
